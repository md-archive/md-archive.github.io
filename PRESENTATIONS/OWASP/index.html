Paragraph A03: What is an Injection?

    "Injection is an attacker's attempt to send data to an application in a way that will change the meaning of commands being sent to an interpreter"

Overview

  • According to OWASP Top 10 List, any XSS or SQL injections are classified as a third most common attack. 94% of Applications were tested for some form of injection with a max incident rate of 19%
  • An average incidence rate of 3% with almost 274 thousands occurences happening around the globe

Description

    Any web application is vulnerable to attack when:

    1. User supplied data is NOT validated, filtered or sanitised by the application

    2. Queries are escaping through a web application interpreter i.e. Executing any kind of queries through a python or javascript interpreters

    3. Hostile data is used within search parameters to extract additional, sensitive records.

    4. Hostile data is directly used or concatenated. The SQL or command contains the structure and malicious data in dynamic queries, commands, or stored procedures.

Common Injection Methods

    Some of the more common injections are:

  • SQL
  • NoSQL
  • OS command
  • Object Relational Mapping (ORM)
  • LDAP
  • Expression Language (EL) or Object Graph Navigation Library (OGNL) injection

Paragraph A04 - Insecure Design

    Nowadays more and more companies and individuals focuse on risks related to design and architectural flaws in IT

    As a community we need to move forward by disciplining ourselves to code or pre-code activities that are critical for the principles of Secure by Design.

Insecure Design Categories

Insecure design is a broad category representing different weaknesses

  • Insecure design is NOT the source of all other Top 10 risk categories.
  • There IS a difference between insecure design and insecure implementation
  • A secure design can still have implementation defects leading to vulnerabilities that may be exploited (Zero Day)
  • One of the factors that contribute to insecure design is the lack of business profiling inherent in the software or system being developed (e.g. using Mac Mini instead of sophisticated Server to distribute data)

Requirements and Resource Management

  • We need to collect and negotiate the requirements for an application of different types with the business, including the protection measures and Good Practices such as Confidentiality, Integrity, Availibility

    • and Authenticity of all data assests and the expected logic

  • A secure design can still have implementation defects leading to vulnerabilities that may be exploited (Zero Day)
  • One of the factors that contribute to insecure design is the lack of business profiling inherent in the software or system being developed (using Mac Mini instead of sophisticated Server to distribute data)

Lifecycle

  • Any software with security in mind requires a secure developement lifecycle, some form of secury design pattern, road methodology, toolong and threat modeling
